# Next.js 架构评审报告

## 总体评分

| 维度               | 评分（1-10） | 简评                                                     |
| ------------------ | ------------ | -------------------------------------------------------- |
| 目录结构合理性     | 9            | Route Groups 划分清晰，层级合理（最深 3 层）             |
| Server/Client 划分 | 8            | Layout 全为 SC，`use client` 使用合理，少量可优化空间    |
| 数据获取模式       | 9            | 全部走 DB 层 + Server Actions，Promise.all 并行化做得好  |

## 高优先级问题（影响性能或安全）

### 1. 全站无 error.tsx 错误边界

- **问题描述**：整个项目 0 个 `error.tsx` 文件——根路由、`(site)` 组、`admin` 组和动态路由 `blog/[slug]` 均无错误边界。任何未捕获异常将展示 Next.js 默认错误页面，用户体验差。
- **问题文件**：`src/app/` 下所有 17 个路由段
- **风险**：可维护性 + 用户体验
- **优化方案**：至少在 `(site)` 和 `admin` 两个 Route Group 级别各添加 `error.tsx`，动态路由 `blog/[slug]` 也应单独处理。
- **优化示例**：

```tsx
// src/app/(site)/error.tsx
'use client'

export default function SiteError({
  error,
  reset
}: {
  error: Error & { digest?: string }
  reset: () => void
}) {
  return (
    <div className="flex min-h-[50vh] flex-col items-center justify-center gap-4">
      <h2>Something went wrong</h2>
      <button onClick={reset}>Try again</button>
    </div>
  )
}
```

### 2. 全站无 loading.tsx 加载状态

- **问题描述**：0 个 `loading.tsx` 文件。数据库查询耗时时（如留言板、相册），页面将白屏等待，无 Suspense 边界。
- **问题文件**：`src/app/` 下所有路由段
- **风险**：性能（感知加载时间）
- **优化方案**：在数据量大的页面（`message`、`photos`、`blog`、`admin/*`）添加 `loading.tsx`，利用已有的 `skeleton.tsx` 组件。
- **优化示例**：

```tsx
// src/app/(site)/blog/loading.tsx
import { Skeleton } from '@/components/ui/skeleton'

export default function BlogLoading() {
  return (
    <div className="space-y-4 p-6">
      {Array.from({ length: 6 }).map((_, i) => (
        <Skeleton key={i} className="h-32 w-full rounded-lg" />
      ))}
    </div>
  )
}
```

### 3. 无 next/dynamic 懒加载大型第三方库

- **问题描述**：`framer-motion` 在 14+ 个文件中被引用，`@dnd-kit` 和 `@tanstack/react-table` 在 Admin 页面使用，均为静态 import，增大首屏 JS bundle。项目中 0 处使用 `next/dynamic`。
- **问题文件**：`src/views/photos/Photos.tsx`、`src/views/home/Gallery.tsx`、`src/components/Animated/Animated.tsx` 等
- **风险**：性能（JS bundle 体积）
- **优化方案**：对仅在特定页面使用的重型组件使用 `dynamic()` 懒加载。注意 `Background.tsx` 已使用 `React.lazy`（做法正确但不一致——建议统一用 `next/dynamic`）。
- **优化示例**：

```tsx
// 优化前 - src/views/photos/Photos.tsx
import { AnimatePresence, motion } from 'framer-motion'
// 优化后
import dynamic from 'next/dynamic'
const PhotoGallery = dynamic(() => import('./PhotoGallery'), { ssr: false })
```

## 中优先级问题（影响可维护性）

### 4. `any` 类型使用（17 处）

- **问题描述**：`src/icons/_internal/hooks.ts` 中 12 处事件处理器参数使用 `any` 类型，`src/views/about/About.tsx` 中 2 处数组遍历使用 `any`，`src/lib/posts.ts` 1 处 `Record<string, any>`。
- **问题文件**：
  - `src/icons/_internal/hooks.ts:10-15, 48-68` — 事件处理器全部为 `(e: any)`
  - `src/views/about/About.tsx:107, 113` — `(group: any)`、`(item: any)`
  - `src/components/Form/types.ts:17` — `defaultValue?: any`
  - `src/components/DataTable/types.ts:31` — `render?: (value: any, ...)`
- **风险**：类型安全
- **优化方案**：

```tsx
// 优化前 - src/icons/_internal/hooks.ts
onMouseEnter: (e: any) => void

// 优化后
onMouseEnter: (e: React.MouseEvent) => void
onTouchStart: (e: React.TouchEvent) => void
```

### 5. Admin CRUD 页面模式重复

- **问题描述**：6 个 Admin 页面（Messages 320 行、Users 344 行、Photos 279 行、Projects 254 行、Playlist 244 行、Poems 205 行）遵循几乎相同的模式：表格列定义 -> 表单字段定义 -> useState 管理 Modal -> CRUD 操作函数 -> DataTable 渲染。重复率约 60%。
- **问题文件**：`src/views/admin/` 下 6 个文件
- **风险**：可维护性（每次修改需同步 6 处）
- **优化方案**：提取通用 CRUD 页面工厂函数或高阶组件：

```tsx
// 可考虑抽象 useCrudPage hook
function useCrudPage<T>({
  createAction,
  updateAction,
  deleteAction
}: CrudActions<T>) {
  const [editingItem, setEditingItem] = useState<T | null>(null)
  const [isModalOpen, setIsModalOpen] = useState(false)
  // ... 统一状态管理逻辑
}
```

### 6. message/page.tsx 中遗留注释代码

- **问题描述**：`src/app/(site)/message/page.tsx:10-21` 存在一大段被注释掉的旧代码。
- **问题文件**：`src/app/(site)/message/page.tsx:10-21`
- **风险**：代码整洁
- **优化方案**：直接删除注释代码块，Git 历史中可追溯。

### 7. dashboard.ts 未遵循 DB 类模式

- **问题描述**：`src/db/dashboard.ts` 是 DB 层中唯一没有使用类封装和 `serialize()` 方法的文件，直接导出函数并返回原始 Prisma 对象（含 Date 类型），与其他 DB 类的 ISO 字符串序列化约定不一致。
- **问题文件**：`src/db/dashboard.ts` 整文件
- **风险**：一致性，Date 序列化可能在 Client Component 中引发问题
- **优化方案**：改为 `DashboardDb` 类，添加 serialize 逻辑，或至少对 `recentMessages` / `recentUsers` 的日期字段做 ISO 转换。

### 8. .env.example 中存在代码未引用的环境变量

- **问题描述**：`BETTER_AUTH_URL` 和 `BETTER_AUTH_API_KEY` 在 `.env.example` 中列出，但源码中从未引用（`BETTER_AUTH_URL` 可能由 better-auth 库自动读取，但 `BETTER_AUTH_API_KEY` 可能确实多余）。`DIRECT_URL` 仅在 `prisma.config.ts` 中使用（正常）。
- **问题文件**：`.env.example:8-9`
- **风险**：低——但可能误导开发者
- **优化方案**：确认 better-auth 是否自动读取这些变量，如不需要则从 `.env.example` 中移除并添加注释说明。

## 低优先级 / 建议改进

### 9. 缺少全局 not-found.tsx 自定义 404 页面

- **问题文件**：`src/app/` 根目录
- **建议**：添加自定义 `not-found.tsx` 以匹配网站整体视觉风格。

### 10. blog/[slug] 动态路由缺少 generateMetadata

- **问题文件**：`src/app/(site)/blog/[slug]/page.tsx`
- **建议**：为博客详情页添加 `generateMetadata`，根据文章标题和描述生成 OpenGraph 元数据，提升 SEO 和社交分享效果。

### 11. Background.tsx 使用 React.lazy 而非 next/dynamic

- **问题文件**：`src/layouts/Background.tsx:5-6`
- **建议**：在 Next.js 项目中统一使用 `next/dynamic` 替代 `React.lazy`，以获得更好的 SSR 控制和一致性。

### 12. 部分数据页面缺少 revalidate 策略

- **问题描述**：只有 `photos`、`project`、`feed.xml` 设置了 `revalidate = 3600`，`message`、`poems`、`blog` 等页面依赖默认行为。
- **问题文件**：`src/app/(site)/message/page.tsx`、`src/app/(site)/poems/page.tsx` 等
- **建议**：根据数据更新频率为各页面配置合适的 revalidate 策略。

## 值得保留的优秀实践

1. **数据获取架构堪称教科书**：Page（RSC）-> Server Action / DB 层 -> Prisma -> PostgreSQL，数据流单向清晰。`Promise.all` 并行化做到位（首页并行请求 playlist + posts，Dashboard 一次性并行 10 个查询）。

2. **Server/Client 边界划分精准**：3 个 Layout 全部是 Server Component，数据在 Page 层获取后通过 Props 传递给 Client View，杜绝了 Client 端直接访问数据库。

3. **DB 层抽象设计优雅**：类封装 + `serialize()` 统一 Date -> ISO 转换，保证 JSON 序列化安全。Prisma 完全被隔离在 `src/db/` 和 `src/lib/prisma.ts` 中。

4. **Server Actions 高度一致性**：6 个 action 文件 100% 遵循 `try/catch` + `actionSuccess/actionError` + `revalidatePath` + `checkAdmin` 模式。

5. **i18n 完备度高**：10 个命名空间，中英文翻译键完全对齐，Server/Client 端各有独立初始化入口。

6. **Page 文件极度精简**：所有 `page.tsx` 只做数据获取和组件委托，最长不超过 20 行，严格遵循"薄 Page"原则。

7. **背景组件的懒加载设计**：`Background.tsx` 使用 `lazy` + `Suspense` 按需加载 Canvas 动画，避免首屏加载大型绘图代码。

8. **已适配 Next.js 15+ 参数类型**：`blog/[slug]/page.tsx` 中 `params` 正确处理为 `Promise<{ slug: string }>` 类型。
