大多数 Next.js 的 i18n 教程都假设你需要 URL-based 的区域路由：`/en/about`、`/zh/about` 等。这意味着你需要把整个应用塞进一个 `[locale]` 动态段，改写每个页面文件，还可能破坏已有链接。

但如果你的个人博客只是想加一个语言切换按钮呢？URL 不变，语言偏好存在 Cookie 里，也没有 SEO 额外开销。这正是本文要讨论的方案。

## 方案原理：Cookie-based i18n

不使用 URL 前缀，我们将用户的语言偏好存在 Cookie 中。数据流如下：

```txt
浏览器请求
    │
    ▼
proxy.ts（Next.js 16 请求拦截器）
  ├── 读取 Cookie（i18n_lang）
  ├── 降级：解析 Accept-Language 请求头
  └── 注入自定义请求头：x-i18n-lang = "zh"
    │
    ├──────────────────────┐
    ▼                      ▼
服务端组件              客户端组件
  getT()                  i18next + LanguageDetector
  读取请求头              读取 Cookie
  创建实例                全局单例
  加载 JSON               动态 import JSON
  返回 { t, lang }       useTranslation() → t()

语言切换 → 写入 Cookie → router.refresh() → 全量重新渲染
```

**核心原则：**

- 服务端从**自定义请求头**（由 proxy 注入）读取语言
- 客户端从**Cookie** 读取语言（通过 LanguageDetector）
- 语言切换：**写入 Cookie + `router.refresh()`**
- URL **永不改变** — `/blog` 永远是 `/blog`

## Cookie 方案 vs URL 方案

取舍非常直接：

- **URL 方式**（`/en/about`、`/zh/about`）：搜索引擎友好（hreflang、独立索引页面），但需要 `[locale]` 目录重构、全局 middleware 守卫、破坏已有链接。
- **Cookie 方式**：不需要任何路由改动，一个 URL 对应所有语言，但搜索引擎只能看到一种语言版本。无法使用 `hreflang` 标签。

对于个人博客或内部工具，这通常不是问题。

本教程假设你已具备：

## 共享配置与类型安全

为 i18n 设置创建唯一的配置中心，服务端和客户端共用。

```ts title="src/i18n/settings.ts"
export const LANGUAGES = ['en', 'zh'] as const
export type Language = (typeof LANGUAGES)[number]
export type Namespace = keyof CustomTypeOptions['resources']

export const FALLBACK_LNG: Language = 'en'
export const DEFAULT_NS: Namespace = 'common'
export const COOKIE_NAME = 'i18n_lang'
export const HEADER_NAME = 'x-i18n-lang'

export function getI18nOptions(
  lng: Language = FALLBACK_LNG,
  ns: Namespace = DEFAULT_NS
) {
  return {
    supportedLngs: LANGUAGES as unknown as string[],
    fallbackLng: FALLBACK_LNG,
    lng,
    fallbackNS: DEFAULT_NS,
    defaultNS: DEFAULT_NS,
    ns,
    interpolation: {
      escapeValue: false // React handles XSS protection
    }
  }
}
```

`Namespace` 定义为 `keyof CustomTypeOptions['resources']`，意味着它自动从你的 `i18next.d.ts` 类型增强中推导。当你添加新翻译 JSON 文件时，只需在 `i18next.d.ts` 加一行 `typeof import(...)`，`Namespace` 类型和泛型 `getT<N extends Namespace>()` 会自动识别。

```ts title="src/types/i18next.d.ts"
import 'i18next'

declare module 'i18next' {
  interface CustomTypeOptions {
    defaultNS: 'common'
    resources: {
      common: typeof import('@/locales/en/common.json')
      home: typeof import('@/locales/en/home.json')
      about: typeof import('@/locales/en/about.json')
      blog: typeof import('@/locales/en/blog.json')
      project: typeof import('@/locales/en/project.json')
      message: typeof import('@/locales/en/message.json')
      poems: typeof import('@/locales/en/poems.json')
      photos: typeof import('@/locales/en/photos.json')
      use: typeof import('@/locales/en/use.json')
      auth: typeof import('@/locales/en/auth.json')
    }
  }
}
```

添加新 namespace — 只需两步：

1. 创建 `src/locales/en/{name}.json` 和 `src/locales/zh/{name}.json`
2. 在 `i18next.d.ts` 加 `typeof import(...)` 行

搞定。无需手动修改 union type。

## proxy.ts — 语言检测的入口

在 Next.js 16 中，`proxy.ts` 替代了旧的 `middleware.ts`。它是一个请求拦截器（不是 WebSocket ���理也不是构建时代理），在每个匹配的请求之前执行。

我们将语言检测逻辑放在**任何**其他逻辑之前。

```ts title="src/proxy.ts"
import type { NextRequest } from 'next/server'

import type { Language } from '@/i18n/settings'

import acceptLanguage from 'accept-language'
import { NextResponse } from 'next/server'

import {
  COOKIE_NAME,
  FALLBACK_LNG,
  HEADER_NAME,
  LANGUAGES
} from '@/i18n/settings'

acceptLanguage.languages([...LANGUAGES])

export async function proxy(request: NextRequest) {
  const { pathname } = request.nextUrl

  // 1. Language detection: Cookie → Accept-Language → fallback
  let lang = request.cookies.get(COOKIE_NAME)?.value

  if (!lang || !LANGUAGES.includes(lang as Language)) {
    lang =
      acceptLanguage.get(request.headers.get('Accept-Language')) ?? FALLBACK_LNG
  }

  const requestHeaders = new Headers(request.headers)
  requestHeaders.set(HEADER_NAME, lang)

  // ← Other logic (auth guards, redirects, etc.) goes below

  return NextResponse.next({
    request: { headers: requestHeaders }
  })
}

export const config = {
  matcher: [
    '/((?!api|_next/static|_next/image|favicon.ico|icons|images|fonts).*)'
  ]
}
```

`config.matcher` 需覆盖所有页面路由，同时排除静态资源和 API 路由。

语言优先级为 **Cookie → Accept-Language 请求头 → fallback `en`**。显式的范围校验（`!LANGUAGES.includes(...)`）防止用户可控值泄露到请求��中。检测完成后，proxy 注入一个自定义请求头（`x-i18n-lang`），服务端组件可通过 `headers()` API 读取。

## 服务端翻译 — getT()

服务端组件无法使用 React hooks，因此需要一个异步工具函数。每次 `getT()` 调用都会创建一个全新的 i18next 实例，防止服务端并发请求间的语言状态泄漏。

```ts title="src/i18n/server.ts"
import type { Language, Namespace } from './settings'

import { createInstance } from 'i18next'
import resourcesToBackend from 'i18next-resources-to-backend'
import { headers } from 'next/headers'
import { initReactI18next } from 'react-i18next/initReactI18next'

import { FALLBACK_LNG, getI18nOptions, HEADER_NAME } from './settings'

async function initI18next(lng: Language, ns: Namespace) {
  const instance = createInstance()
  instance.use(initReactI18next)
  instance.use(
    resourcesToBackend(
      (language: string, namespace: string) =>
        import(`@/locales/${language}/${namespace}.json`)
    )
  )
  await instance.init(getI18nOptions(lng, ns))
  return instance
}

/**
 * Get the translation function for Server Components and generateMetadata.
 *
 * Every call creates a fresh instance to prevent language state leaking
 * between concurrent requests on the server.
 *
 * @example
 * const { t, lang } = await getT('about')
 * return <h1>{t('title')}</h1>
 */
export async function getT<N extends Namespace = 'common'>(
  ns: N = 'common' as N
) {
  const headerList = await headers()
  const lang = (headerList.get(HEADER_NAME) as Language) ?? FALLBACK_LNG
  const i18next = await initI18next(lang, ns)

  return {
    t: i18next.getFixedT(lang, ns),
    lang,
    i18next
  }
}
```

泛型 `N extends Namespace` 确保 namespace 参数在编译时即可被类型检查。`getFixedT(lang, ns)` 直接接收 namespace — i18next 同时支持 `string` 和 `string[]` 两种形式。

## 根布局 — 动态 `lang` 属性

根布局读取 proxy 注入的请求头来设置 `<html lang>` 和 `generateMetadata`：

```tsx title="src/app/layout.tsx"
import type { Metadata } from 'next'

import { headers } from 'next/headers'

import { ROBOTO_MONO, AUDIO_WIDE } from '@/styles/global.css'
import { FALLBACK_LNG, HEADER_NAME, getT } from '@/i18n/settings'

export async function generateMetadata(): Promise<Metadata> {
  const { t } = await getT('common')

  return {
    title: {
      default: 'King3',
      template: '%s - King3'
    },
    description: t('metadata.root.description'),
    icons: {
      icon: '/icons/favicon.svg'
    }
  }
}

export default async function RootLayout({
  children
}: {
  children: React.ReactNode
}) {
  const headerList = await headers()
  const lang = headerList.get(HEADER_NAME) ?? FALLBACK_LNG

  return (
    <html
      lang={lang}
      suppressHydrationWarning
      className={`${robotoMono.variable} ${audioWide.variable}`}
    >
      <body className="min-h-screen scroll-smooth antialiased">
        {/* Providers ... */}
        {children}
      </body>
    </html>
  )
}
```

`generateMetadata()` 和 `RootLayout` 都读取同一个请求头，所以 `<html lang>`、`<meta description>` 和渲染的页面文本全部同步。

## 客户端初始化

客户端组件使用全局 i18next 单例配合 `LanguageDetector` 来从 Cookie 自动检测语言。

```ts title="src/i18n/client.ts"
'use client'

import i18next from 'i18next'
import LanguageDetector from 'i18next-browser-languagedetector'
import resourcesToBackend from 'i18next-resources-to-backend'
import {
  initReactI18next,
  useTranslation as useI18nextTranslation
} from 'react-i18next'

import { COOKIE_NAME, getI18nOptions, LANGUAGES } from './settings'

const runsOnServer = typeof window === 'undefined'

i18next
  .use(initReactI18next)
  .use(LanguageDetector)
  .use(
    resourcesToBackend(
      (language: string, namespace: string) =>
        import(`@/locales/${language}/${namespace}.json`)
    )
  )
  .init({
    ...getI18nOptions(),
    lng: undefined, // Let LanguageDetector decide
    detection: {
      order: ['cookie', 'navigator'],
      lookupCookie: COOKIE_NAME,
      caches: ['cookie']
    },
    // During SSR, preload all languages so hydration output matches.
    // During CSR, leave empty — languages load dynamically on demand.
    preload: runsOnServer ? (LANGUAGES as unknown as string[]) : []
  })

const useTranslation = useI18nextTranslation

export { useTranslation }
export default i18next
```

`lng: undefined` 非常关键 — 这告诉 i18next 完全依赖 LanguageDetector 来决定语言，而不是使用硬编码值。

`useTranslation` 别名（`const useTranslation = useI18nextTranslation`）让所有组件从统一路径（`@/i18n/client`）导入，无论单例���否已经初始化。这是 hook 的唯一入口点。

## 语言切换组件

一个处理 Cookie 持久化、客户端 i18next 更新和服务端重新渲染的组件。

```tsx title="src/components/LanguageSwitcher.tsx"
'use client'

import type { Language } from '@/i18n/settings'
import type { InteractiveIconProps, SvgIcon } from '@/icons'

import { useRouter } from 'next/navigation'
import { useEffect, useState } from 'react'

import { useTranslation } from '@/i18n/client'
import { COOKIE_NAME } from '@/i18n/settings'
import { InteractiveIcon, LanguageIcon } from '@/icons'

export function LanguageSwitcher({
  size,
  color,
  strokeWidth,
  ...delegated
}: InteractiveIconProps<SvgIcon>) {
  const router = useRouter()
  const { i18n, t } = useTranslation()
  const currentLang = (i18n.language ?? 'en') as Language
  const [mounted, setMounted] = useState(false)

  useEffect(() => setMounted(true), [])

  const switchLang = (event: React.MouseEvent) => {
    event.preventDefault()
    const nextLang: Language = currentLang === 'en' ? 'zh' : 'en'
    const maxAge = 60 * 60 * 24 * 365
    document.cookie = `${COOKIE_NAME}=${nextLang};path=/;max-age=${maxAge};SameSite=Lax`
    i18n.changeLanguage(nextLang)
    router.refresh()
  }

  // SSR phase: render a static icon to avoid hydration mismatch
  if (!mounted) {
    return (
      <InteractiveIcon {...delegated} trigger="hover">
        {() => (
          <LanguageIcon
            isEN
            isHovered={false}
            {...{ size, color, strokeWidth }}
          />
        )}
      </InteractiveIcon>
    )
  }

  const isEN = currentLang === 'en'

  return (
    <InteractiveIcon
      {...delegated}
      alt={t('localeSwitcher.label')}
      trigger="hover"
      onClick={switchLang}
    >
      {({ isHovered }) => (
        <LanguageIcon
          isEN={isEN}
          isHovered={isHovered}
          {...{ size, color, strokeWidth }}
        />
      )}
    </InteractiveIcon>
  )
}
```

语言切换三步骤：

1. **`document.cookie`** — 让 proxy 在下一次请求时检测到新语言
2. **`changeLanguage()`** — 让客户端组件立即更新
3. **`router.refresh()`** — 让服务端组件用新语言重新渲染

SSR 阶段组件不知道用户的语言（首次访问还不存在 cookie）。如果没有 `mounted` 保护，服务器端会按 `isEN = true` 渲染，而客户端可能检测到 `zh`，导致 hydration 不匹配。`mounted` 状态确保客户端在 hydration 完成后重新渲染为正确语言。

## Hydration 一致性

i18n 最常见的坑之一就是 hydration mismatch — 服务端渲染了 A 语言而客户端检测到了 B 语言。在这个方案中，服务端和客户端都从**同一个 Cookie** 读取：

- **服务端**：proxy 读取 `i18n_lang` Cookie → 注入 `x-i18n-lang` 请求头 → `getT()` 读取
- **客户端**：LanguageDetector 读取 `i18n_lang` Cookie → 初始化 i18next

因为两端都从同一个 Cookie 值派生，所以天然一致。

## 首次访问行为

当用户在第一次访问（还没有 Cookie）时：

1. Proxy 从浏览器读取 `Accept-Language` 请求头
2. 如果浏览器语言为中文即检测 `zh`
3. 否则 fallback 到 `en`
4. 切换语言后 cookie 保存选择

## 按需加载

`i18next-resources-to-backend` 插件通过动态 `import()` 按需加载翻译文件：

- 访问首页 → 加载 `common.json` + `home.json`
- 访问 about → 加载 `common.json` + `about.json`
- 没访问过的页面翻译文件永远不会被下载
